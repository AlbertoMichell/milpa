**NOTA IMPORTANTE - ¿Qué es el "harness"?**
El harness (arnés de evaluación) es una herramienta que mide qué tan bien funciona el sistema RAG. Toma dos archivos: uno con las respuestas correctas esperadas ("gold") y otro con las respuestas que dio el sistema ("pred"), luego calcula métricas numéricas (precision, recall, nDCG, MRR, faithfulness) que indican qué tan preciso, completo y ordenado fue el resultado. Es como un examen automático para el sistema de búsqueda.

────────────────────────────────────────────────────────────────

En este bloque incorporé tres piezas clave y las dejé operando punta a punta: 1) un harness de evaluación para medir con rigor la calidad del RAG, 2) una capa de sanitización y clic-through seguro en el Presenter v1 con headers de seguridad endurecidos, y 3) una página de verificación visual (/ui/checks) que permite comprobar gráficamente el estado de servicios, la sanitización de HTML y la validación de parámetros de clic-through. Con esto, el sistema puede medir precisión de recuperación (precision@k, recall@k, MRR, nDCG), cobertura de entidades y, cuando aplique, fidelidad de la síntesis; al mismo tiempo, cualquier respuesta HTML servida por el Presenter es sanitizada para permitir únicamente enlaces internos seguros.

El harness es auto-contenido, CLI-friendly y opera sobre archivos JSONL de “oro” y “predicción” alineados por query. Calcula métricas a nivel consulta y promedia: precision@k y recall@k para diferentes ks configurables, MRR para la primera coincidencia relevante, nDCG@k como ranking-awareness, entity_coverage@k si hay entidades por consulta y, opcionalmente, faithfulness si la predicción la aporta. Incluí ejemplos mínimos para reproducir resultados y un test de humo que valida rangos esperables.

En el Presenter v1 añadí un plugin de sanitización que aplica una política de allowlist estricta (solo a, em, strong, p, ul/ol/li, br, code, pre) y bloquea esquemas no permitidos (http, https, mailto, etc.). Toda URL externa se reescribe automáticamente a “#”, mientras que los enlaces internos (rutas locales, anchors o query params) se preservan y permiten clic-through seguro con metadatos controlados (data-cite, data-page, data-bbox, data-table, data-row, data-col). Endurecí cabeceras de seguridad (CSP restrictiva, HSTS, X-Frame-Options=DENY, Referrer-Policy=no-referrer) y mantuve rate-limiting. Para facilitar validación integral, agregué rutas de demo/preview y una página gráfica de checks.

La página /ui/checks centraliza pruebas manuales rápidas sin herramientas externas: muestra el estado del Presenter y del Backend de IA, permite cargar un HTML de ejemplo sanitizado por el servidor, probar sanitización sobre entradas personalizadas y validar el endpoint de clic-through con parámetros válidos e inválidos. Ahora incluye una tarjeta de “Guía y propósito”, etiquetas de “Lectura” y “Salida” en cada panel, vistas de “Renderizado” y “HTML como texto” para la sanitización, y un botón “Actualizar” para reconsultar el estado. Esta página facilita QA y reduce tiempos de verificación en entornos con acceso limitado a utilidades de consola.

SEGURIDAD Y CABECERAS

• CSP: default-src 'none', connect-src 'self', style-src 'self' 'unsafe-inline', img-src 'self' data:, frame-ancestors 'none', base-uri 'none', form-action 'self'. Bloquea recursos de terceros por defecto.
• HSTS: Strict-Transport-Security habilitado (efectivo en HTTPS) para forzar tráfico seguro en navegadores.
• X-Frame-Options: DENY para evitar clickjacking.
• Referrer-Policy: no-referrer para no filtrar orígenes.
• Rate limiting: 60 req/min por IP en el Presenter.
• Sanitización: allowlist de tags/attrs mínima; allowedSchemes vacío para impedir http/https/mailto; enlaces externos reescritos a “#”.

• La CSP estricta bloquea carga de recursos de terceros; ajustar ALLOWED_ORIGINS en caso de UI externa.
• HSTS solo tiene efecto en HTTPS; en local HTTP los navegadores lo ignoran.
