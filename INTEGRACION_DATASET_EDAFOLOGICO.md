# Integración del Dataset Edafológico

## Descripción General

El sistema integra el dataset edafológico sintético dataset_con_awc.xlsx que contiene 360 registros de propiedades del suelo distribuidos en 120 sitios de tres localidades de Veracruz (Coacoatzintla, Xalapa, Naolinco), con mediciones en tres profundidades por sitio (0-5cm, 5-15cm, 15-30cm). El dataset comprende 26 variables sin valores nulos que describen propiedades físicas, químicas y de capacidad hídrica del suelo. Su propósito es proporcionar datos reales que contextualizan las recomendaciones agronómicas del sistema RAG, permitiendo adaptar recomendaciones documentales generales a condiciones edafológicas específicas de cada localidad.


## Clasificación de Variables

El dataset contiene 26 variables clasificadas en cinco grupos funcionales según su naturaleza y variabilidad temporal.

**Variables de Identificación (7 variables)**: site_id identifica cada sitio de muestreo con valores de 1 a 120. localidad es categórica nominal con tres valores (Coacoatzintla, Xalapa, Naolinco) y se usa como filtro principal en el frontend. Las coordenadas lat y lon son numéricas continuas que definen la posición geográfica de cada sitio (19.516-19.680°N, -96.961 a -96.836°O). fecha es temporal tipo datetime64 que registra el momento del muestreo (2024-01-02 a 2024-12-20). depth_cm_from y depth_cm_to son enteros ordinales que definen los tres horizontes de profundidad. Estas variables son fijas excepto fecha.

**Variables Físicas del Suelo (6 variables)**: humedad_suelo_vv es numérica continua que varía entre 0.05 y 0.45 (media 0.25), representa el contenido volumétrico de agua y es dinámica con variación temporal diaria y estacional, usada en gráficos de series temporales del frontend. temperatura_suelo_C oscila entre 10°C y 32.3°C (media 23.6°C) y también es dinámica. arena_pct, limo_pct y arcilla_pct son porcentajes que caracterizan la textura del suelo (rangos 1.4-84.6%, 1.6-83.8%, 2.0-83.7% respectivamente) y son fijas o cuasi-permanentes. densidad_aparente_g_cm3 varía entre 1.34 y 1.65 g/cm³ (media 1.52) y cambia lentamente con el manejo, se usa para calcular AWC.

**Variables Químicas del Suelo (8 variables)**: ph_h2o oscila entre 4.82 y 8.26 (media 6.41) y controla la disponibilidad de nutrientes, es semi-dinámica modificable mediante encalado. conductividad_electrica_dS_m mide salinidad (0.05-2.41, media 0.92). soc_g_kg es carbono orgánico (5-35.1, media 18.6) y materia_organica_pct es su equivalente porcentual (0.5-6.32%, media 3.2%), ambas semi-dinámicas que indican calidad del suelo. n_total_g_kg, p_disponible_mg_kg y k_disponible_mg_kg representan nutrientes principales (rangos 0.2-2.28, 3-52.7, 50-365.3 respectivamente) y se visualizan en el gráfico de nutrientes del frontend, son semi-dinámicas y responden a fertilización. cec_cmolc_kg es capacidad de intercambio catiónico (2.4-34.3, media 15.6) y es cuasi-permanente.

**Variables de Capacidad Hídrica (4 variables)**: cc_pct es capacidad de campo (45.1-101.4%, media 64.4%) y pm_pct es punto de marchitez permanente (7.8-50.2%, media 23.5%), ambas fijas. profundidad_efectiva_cm toma valores 5, 10 o 15 según el horizonte. awc_mm_m es agua disponible calculada como (CC - PM) × profundidad_efectiva × 10, varía entre 24.6 y 119.2 mm/m (media 62.0) y es la variable crítica para planificación de riego en el frontend.

**Variable de Salida (1 variable)**: cultivo_sugerido es categórica nominal con tres valores (Maíz, Frijol, Calabaza) que representa la recomendación agronómica derivada de todas las propiedades edafológicas. Se usa como target en modelos de clasificación y como filtro en el frontend para mostrar sitios según aptitud de cultivo.

Respecto a variabilidad temporal, las variables se clasifican en fijas (textura, coordenadas, densidad, CEC, capacidades hídricas), semi-dinámicas (pH, nutrientes, materia orgánica que cambian con manejo agronómico), y dinámicas (humedad y temperatura que varían diaria y estacionalmente). Las categóricas son localidad y cultivo_sugerido. Las numéricas continuas suman 22 variables de tipo float64 o int64.


## Flujo de Datos e Integración con el Sistema

El dataset ingresa al sistema desde el archivo Excel dataset_con_awc.xlsx ubicado en el directorio Instruccion. El backend Python usa pandas para leer el archivo mediante read_excel() y cargarlo en un DataFrame en memoria. Durante la carga se validan tipos de datos, rangos de valores plausibles y ausencia de valores nulos. El DataFrame permite operaciones de filtrado por localidad o fecha, agregación espacial para calcular AWC total sumando los tres horizontes de cada sitio, y cálculo de variables derivadas como relación carbono-nitrógeno o índices texturales. Para modelos de machine learning, las variables categóricas como localidad se codifican mediante one-hot encoding creando variables binarias por categoría, y las numéricas se normalizan para equiparar escalas.

Los datos procesados se exponen al frontend mediante una API REST implementada en el Presenter de Node/TypeScript que retorna respuestas en formato JSON. Los endpoints principales incluyen uno para series temporales de humedad que acepta parámetros de localidad y rango de días y retorna arrays de fechas y valores de humedad, otro para nutrientes de un sitio específico que retorna valores de N, P, K con su estado (bajo/óptimo/alto), uno para recomendación de cultivo por localidad que retorna el cultivo sugerido con sitios similares y AWC promedio, y uno para búsqueda de sitios similares que recibe propiedades del suelo del usuario y retorna los K sitios más cercanos usando distancia euclidiana normalizada.

El frontend en datos.html consume estos endpoints mediante peticiones asíncronas con fetch() o axios(). El gráfico de humedad del suelo implementado con Chart.js actualiza dinámicamente la serie temporal cuando el usuario cambia la localidad o el período de tiempo seleccionado. El gráfico de nutrientes muestra N, P, K mediante barras o radar y actualiza barras de progreso HTML con colores que indican el estado de cada nutriente respecto a rangos óptimos. Un mapa interactivo con Leaflet usa las coordenadas lat y lon para posicionar marcadores de sitios, coloreados según el cultivo sugerido, permitiendo al usuario explorar patrones espaciales de aptitud de cultivos.

La integración con el sistema RAG ocurre de forma sinérgica donde datos numéricos complementan conocimiento textual. Cuando un usuario consulta algo como qué fertilizante necesita para maíz en pH 5.5, el RAG recupera fragmentos de documentos técnicos usando BM25 y BERT que proporcionan recomendaciones generales, mientras simultáneamente el backend filtra el dataset por cultivo igual a Maíz y pH entre 5.0 y 6.0, extrayendo estadísticas de nutrientes de sitios similares. El proceso de enriquecimiento combina fragmentos documentales con métricas del dataset, por ejemplo si un documento recomienda 50 kg/ha de nitrógeno pero los datos muestran que sitios similares requirieron 80 kg/ha para buen rendimiento, la recomendación se ajusta contextualmente. El sistema también valida recomendaciones documentales contra observaciones del dataset, emitiendo alertas si hay inconsistencias, y usa K-NN sobre variables clave para identificar sitios de referencia análogos al del usuario, reportando prácticas exitosas observadas en esos sitios.

Los formatos de datos se transforman según la capa del sistema. El origen es Excel tabular con 360 filas y 26 columnas. En backend Python es un DataFrame tipo diccionario con tipos int64, float64, object para categóricas, y datetime64 para fecha. La API retorna JSON con estructura de objetos anidados conteniendo propiedades del suelo, nutrientes agrupados, y metadatos. El frontend JavaScript maneja arrays simples de números para alimentar gráficos de Chart.js y objetos para actualizar elementos del DOM. Para escalabilidad futura se considera migrar a PostgreSQL para queries indexadas o Parquet para compresión columnar, y usar Redis para caché de resultados frecuentes como promedios por localidad.


## Arquitectura Técnica del Sistema

El stack tecnológico se organiza en cuatro capas principales. La capa de datos utiliza el archivo Excel como fuente original de 360 registros con 26 variables. La capa de backend Python procesa el dataset usando pandas para manipulación de DataFrames, numpy para cálculos vectorizados, scikit-learn para algoritmos de similitud K-NN y normalización, y openpyxl para lectura de archivos Excel. La capa de presentación implementada en Node/TypeScript con framework Fastify o Express maneja endpoints RESTful que retornan JSON con validación de esquemas mediante Zod. La capa de frontend utiliza Chart.js para gráficos de líneas de series temporales de humedad y gráficos de barras o radar para nutrientes, Leaflet para mapas interactivos con marcadores georreferenciados de sitios, y Bootstrap 5 para componentes de interfaz como barras de progreso, tarjetas informativas y pestañas de navegación.

Para persistencia de datos, el MVP actual carga el Excel completo en memoria al iniciar el servidor usando pandas, lo cual es eficiente dado el tamaño modesto del dataset de aproximadamente 100 KB. Para escalabilidad futura con datasets de mayor tamaño se considera migración a PostgreSQL para queries indexadas eficientes, uso de formato Parquet para compresión columnar en almacenamiento, y Redis para caché de resultados frecuentes como promedios por localidad o series temporales de último mes con tiempo de vida de 1 hora.

El sistema maneja transformaciones de formato según la capa. El origen Excel es tabular con 360 filas por 26 columnas. En backend Python el DataFrame es una estructura tipo diccionario con tipos específicos: int64 para identificadores y profundidades, float64 para las 19 variables numéricas continuas, object para variables categóricas como localidad y cultivo_sugerido, y datetime64 para la variable temporal fecha. La API expone datos en JSON con estructura de objetos anidados que agrupan propiedades relacionadas como nutrientes o coordenadas geográficas. El frontend JavaScript consume arrays simples de números para alimentar gráficos y objetos para manipular el DOM. Los datos se transforman mediante serialización JSON entre capas y parsing en el cliente.
