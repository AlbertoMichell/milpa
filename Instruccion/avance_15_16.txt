**ESTADO DE IMPLEMENTACIÓN:** ✅ COMPLETADO Y VERIFICADO

Este sprint (15-16) implementó control de carga robusto en el Presenter mediante scheduler con colas acotadas, circuit breaker y timeouts configurables. El sistema ahora rechaza peticiones de forma controlada cuando está saturado (HTTP 429), previene sobrecarga del backend IA y expone métricas detalladas para monitoreo en tiempo real.

**NOTA IMPORTANTE:** La interfaz web `/ui/checks` y `/ui/library` fueron implementadas en el SPRINT 12-14. Este sprint (15-16) EXTIENDE `/ui/checks` agregando el panel "Estatus de ejecución" que muestra métricas del scheduler/circuit breaker en tiempo real.

────────────────────────────────────────────────────────────────

En este bloque se implementaron tres ejes fundamentales para robustecer la operación en producción y garantizar un servicio confiable bajo cualquier condición de carga. El primer eje asegura el cumplimiento operacional mediante trazabilidad completa que correlaciona cada solicitud con identificadores únicos para auditoría, gobernanza de datos que protege información sensible evitando registrar PII en claro y aplicando máscaras cuando sea necesario, y políticas de retención que establecen ciclos de vida para logs y datos temporales sin acumulación indefinida. El segundo eje implementa control de carga mediante colas acotadas que limitan la concurrencia máxima de peticiones simultáneas al backend de inteligencia artificial, backpressure que rechaza ordenadamente nuevas peticiones cuando el sistema alcanza su capacidad máxima devolviendo códigos HTTP 429 con encabezados Retry-After que indican al cliente cuándo reintentar, y timeouts configurables que previenen peticiones zombis cancelando automáticamente operaciones que excedan umbrales de tiempo establecidos. El tercer eje introduce degradación controlada mediante un circuit breaker que monitorea tasas de error y latencias percentiles, abriendo el circuito cuando detecta que el backend está saturado o no responde para evitar sobrecargarlo con más peticiones mientras se recupera, fallbacks que devuelven respuestas cacheadas previamente almacenadas o plantillas predefinidas cuando la inteligencia artificial no está disponible manteniendo funcionalidad básica, y comunicación transparente al usuario mediante mensajes en la interfaz que explican situaciones de alto tráfico con sugerencias de reintento en lugar de fallos silenciosos.

Con estos tres ejes implementados, el Presenter funciona como una válvula reguladora inteligente entre la aplicación web y el backend de inteligencia artificial, actuando como capa de protección que previene sobrecargas mediante límites de concurrencia y rechazos ordenados, como regulador que mide continuamente latencias y tasas de error ajustando el flujo de peticiones según la capacidad disponible, como observador que expone métricas detalladas en tiempo real sobre tareas en vuelo, profundidad de cola, rechazos por saturación y aperturas de circuito permitiendo detectar problemas antes de que impacten masivamente a usuarios, y como sistema de degradación que mantiene experiencia predecible incluso cuando el backend falla devolviendo respuestas cacheadas o reducidas en lugar de errores catastróficos, garantizando que el sistema pueda operar bajo condiciones adversas sin colapsar completamente.

OBJETIVOS CONCRETOS (15–16):

• Cumplimiento y seguridad operacional:
  - Trazabilidad: correlación de solicitudes con IDs únicos (request-id) y logs estructurados.
  - Protección de datos: no registrar PII en claro; mascaras/anonimización en logs cuando aplique.
  - Retención y rotación: política base (p.ej., 14–30 días) para logs y datos temporales.
  - Controles de acceso: mantener segregación (Web → Presenter → IA) y no exponer la IA directamente.
  - AV y sanitización: conservar escaneo antivirus en subidas y sanitización de HTML end-to-end.

• Colas, backpressure y límites:
  - Concurrencia máxima por proceso (p.ej., MAX_CONCURRENCY=8) para llamadas a la IA.
  - Cola acotada (QUEUE_CAPACITY, p.ej., 64) y política cuando se llena (rechazo inmediato 429 con Retry-After).
  - Timeouts y cancelación (UPSTREAM_TIMEOUT_MS y DEADLINE_MS) para evitar “requests zombis”.
  - Backpressure visible: métricas de en_cola, en_vuelo, latencias y 429/5xx para observabilidad.

• Degradación controlada (graceful degradation):
  - Circuit breaker: abrir circuito si error/latencia supera umbrales (CIRCUIT_OPEN_SECS) y responder rápido.
  - Fallbacks: devolver resultados cacheados/previos o plantillas reducidas cuando la IA no responde a tiempo.
  - Comunicación a usuario: banner/mensaje en UI indicando “alto tráfico” y sugerencia de reintento.

REQUISITOS TÉCNICOS Y ACEPTACIÓN (CUMPLIDOS)

• Requisitos:
  - Límite de concurrencia efectivo en el Presenter para todas las llamadas /ai/*.
  - Cola acotada con tamaño configurable; cuando se alcanza el máximo, se rechaza con 429 y encabezado Retry-After.
  - Timeouts claros: se cancela la solicitud a la IA si excede UPSTREAM_TIMEOUT_MS; se devuelve 504/timeout o fallback.
  - Circuit breaker con ventana de evaluación (tasa de fallo/latencia p95) y enfriamiento.
  - Métricas Prometheus: gauges para en_cola y en_vuelo; histogram para latencias; counters para total, 429, timeouts, circuit_open.
  - Logs listos para correlación (request-id) y outcome (hit/fallback/rejected/timeout), sin PII.

• Criterios de aceptación:
  - Bajo carga sobre el Presenter, se observan rechazos 429 controlados y la latencia se mantiene estable.
  - Cuando la IA está lenta, el Presenter responde en < DEADLINE_MS con fallback o 504 bien formateado.
  - El estado “circuito abierto” se refleja en /runtime/status y en métricas; se cierra tras la ventana de enfriamiento.
  - La UI “Milpa • Verificaciones” muestra en vivo en_vuelo, en_cola, circuito y modo_degradado.

DISEÑO IMPLEMENTADO (A NIVEL PRESENTER)

• Variables de entorno (soportadas):
  - MAX_CONCURRENCY=8
  - QUEUE_CAPACITY=64
  - UPSTREAM_TIMEOUT_MS=9000
  - DEADLINE_MS=10000
  - CIRCUIT_OPEN_SECS=20

• Scheduler:
  - Cola FIFO (bounded). Si la cola está llena al llegar una nueva tarea, rechazar 429 al instante.
  - Un “worker pool” de tamaño MAX_CONCURRENCY que saca tareas de la cola.
  - Cada tarea respeta un abort-signal con deadline; si vence, cancelar y marcar timeout.

• Circuit breaker:
  - Estados: closed → half-open → open.
  - Umbrales: p95 > T o error_rate > X% en ventana N → open durante CIRCUIT_OPEN_SECS.
  - En open: responder 503 rápido con Retry-After; en half-open: permitir M “canarios” para probar recuperación.

• Degradación/fallbacks:
  - Cache ligera (memoria) de la última respuesta “razonable” por consulta normalizada.
  - Resumen/esqueleto predefinido (mensaje amable + pasos siguientes) si no hay cache.
  - Reducir complejidad de la petición a IA (p.ej., bajar k o desactivar reranking) cuando se detecta saturación.

• Observabilidad:
  - /metrics: añadir métricas nuevas (en_cola, en_vuelo, latencias, 429, timeouts, circuit_open_total, fallback_total).
  - /runtime/status: JSON con { en_cola, en_vuelo, circuito: "open|half-open|closed", modo_degradado: bool, capacidad: { max_concurrency, queue_capacity } }.
  - UI /ui/checks: tarjeta “Estatus de ejecución” que consulta /runtime/status y muestra indicadores en vivo.

QUÉ MUESTRA CADA PANEL EN /UI/CHECKS Y POR QUÉ

1) Estado de servicios (� HEALTH):
   - Qué es: Verifica conectividad y disponibilidad del Presenter y del Backend IA.
   - Para qué: Confirmar que ambos servicios responden correctamente antes de cualquier prueba.
   - Qué ver: Status "OK" (verde) indica servicio funcional; "ERROR" (rojo) requiere revisar logs.
   - Por qué: Si algún servicio está caído, todo lo demás fallará. Este es el primer check.

2) Demostración de sanitización ( SECURITY):
   - Qué es: Valida que el HTML generado por el sistema esté limpio y seguro (sin scripts maliciosos, enlaces externos bloqueados).
   - Para qué: Prevenir XSS y garantizar que solo se permiten enlaces internos seguros.
   - Qué ver: Enlaces externos deben quedar como "#", atributos peligrosos eliminados. Se muestra tanto el HTML renderizado como el código fuente escapado.
   - Por qué: Las respuestas del sistema incluyen HTML con fragmentos citados. Sin sanitización, un atacante podría inyectar código malicioso.

3) Prueba manual de sanitización (🧪 SECURITY):
   - Qué es: Sandbox para probar HTML personalizado contra el sanitizador.
   - Para qué: Validar que cualquier HTML, sin importar su origen, pasa por el filtro de seguridad.
   - Qué ver: Prueba inyectar scripts, enlaces externos y atributos maliciosos. Todos deben ser bloqueados.
   - Por qué: Permite al equipo validar casos edge y confirmar que la protección funciona contra vectores de ataque reales.

4) Validación clic-through (� VALIDATION):
   - Qué es: Verifica que las rutas de documentos validen correctamente parámetros (doc_id, page).
   - Para qué: Prevenir inyección de rutas y parámetros maliciosos que podrían explotar el sistema.
   - Qué ver: Rutas válidas (/doc/demo?page=2) devuelven 200 OK. Rutas inválidas (/doc/@@@?page=-1) devuelven 400 Bad Request.
   - Por qué: El clic-through permite navegar a páginas específicas de documentos. Sin validación, podría usarse para ataques de path traversal.

5) Biblioteca (� DATA):
   - Qué es: Lista documentos indexados disponibles para consulta.
   - Para qué: Verificar que el backend IA devuelve correctamente el catálogo con metadatos completos.
   - Qué ver: Cada documento debe tener: nombre, autor, año, tipo, país, idioma, extraído_de. Se muestran en tarjetas con botón "Seleccionar" para ir a la página completa de Biblioteca.
   - Por qué: Valida que la ingestión y extracción de documentos funcionó correctamente y que los datos están accesibles.

6) Estatus de ejecución (⚡ RUNTIME):
   - Qué es: Indicadores en tiempo real del scheduler, cola de peticiones y circuit breaker del Presenter.
   - Para qué: Monitorear carga del sistema, capacidad disponible y prevención de sobrecargas.
   - Qué ver: 
     - en_vuelo: peticiones actualmente en proceso
     - en_cola: peticiones esperando a ser procesadas
     - circuito: estado del circuit breaker (open/half-open/closed)
     - modo_degradado: si el sistema está operando con fallbacks
     - capacidad: límites configurados (max_concurrency, queue_capacity)
   - Por qué: Permite detectar cuellos de botella, saturación y recuperación del sistema en tiempo real.

BIBLIOTECA: PROPÓSITO Y FUNCIONALIDAD

La Biblioteca es el catálogo central de documentos que el sistema ha procesado. Cada documento se muestra con sus metadatos (autor, año, tipo) y permite búsqueda avanzada:

• Búsqueda por contenido: No solo busca en título/autor/fuente, sino que también busca dentro del texto extraído de los documentos (tabla fragments). Si un documento contiene la palabra "Centro" en alguna página, lo encontrará aunque no esté en el título.

• Filtros combinables:
  - Año: desplegable con todos los años disponibles
  - Autor: lista alfabética con buscador inline
  - Modo palabra: activa búsqueda AND (todos los términos deben aparecer)
  - Botón Limpiar: restablece todos los filtros

• Vista de detalle:
  - Muestra metadatos completos en un grid limpio
  - Tablas extraídas del documento con headers y filas
  - Desglose por fila: cada fila se desglosa como "columna: valor" para fácil lectura
  - Vista técnica (JSON) para debugging

• Cómo funcionará en el futuro:
  - El Frontend (aplicación separada de React/Vue) consumirá estos mismos endpoints del Presenter.
  - La UI actual sirve como: 1) demo funcional del API, 2) herramienta de QA para el equipo, 3) referencia visual para el Frontend.
  - Se añadirán filtros por tipo de documento, país, idioma, y búsqueda semántica (vectorial).
  - Se integrará con el RAG: al buscar, se mostrarán fragmentos relevantes resaltados con citas a páginas específicas.

COMANDOS PARA PROBAR LA WEB DEL PRESENTER (PowerShell)

1) Carga completa del stack (desde cero):
  cd C:\milpa
  docker compose down
  docker compose build --no-cache
  docker compose up -d

  # Búsqueda simple
  (Invoke-RestMethod http://localhost:8080/ai/library?q=prueba).items | Select-Object nombre,autor,año
  
  # Búsqueda con filtros
  (Invoke-RestMethod "http://localhost:8080/ai/library?q=Centro&word=true&year=2024").items | Select-Object nombre,autor,año

  Invoke-RestMethod http://localhost:8080/runtime/status

7) Prueba de carga controlada (simular múltiples usuarios):
  # Crear 20 peticiones concurrentes
  $jobs = 1..20 | ForEach-Object { 
    Start-Job -ScriptBlock { 
      Invoke-WebRequest -UseBasicParsing http://localhost:8080/ai/library -TimeoutSec 10 | Out-Null 
    } 
  }
  Wait-Job $jobs
  Receive-Job $jobs | Out-Null
  # Luego verificar métricas
  Invoke-RestMethod http://localhost:8080/runtime/status

8) Solo reconstruir y reiniciar el Presenter (desarrollo rápido):
  cd C:\milpa
  docker compose build presenter
  docker compose up -d presenter
  # Esperar unos segundos
  Start-Sleep -Seconds 5
  # Verificar
  Invoke-RestMethod http://localhost:8080/health

9) Reiniciar sin rebuild (más rápido, solo si no cambiaste código):
  cd C:\milpa
  docker compose restart presenter
  Start-Sleep -Seconds 3
  Invoke-RestMethod http://localhost:8080/health

POR QUÉ AGREGAMOS ESTAS PÁGINAS

• Accesibilidad: No todos en el equipo manejan línea de comandos o herramientas de API. Una interfaz web permite a PM, QA y stakeholders verificar el sistema sin barreras técnicas.

• QA ágil: En lugar de escribir scripts de prueba complejos, se puede abrir el navegador, hacer clic en "Cargar demo" y ver al instante si la sanitización funciona.

• Documentación viva: Cada panel explica qué hace, para qué sirve y qué resultados esperar. Es documentación que se mantiene actualizada porque está integrada en el código.

• Demo funcional: Sirve como referencia visual para el equipo de Frontend. En lugar de documentar el API en texto, pueden ver y usar la UI del Presenter como ejemplo funcional.

• Debugging rápido: Si algo falla en producción, se puede abrir /ui/checks y ver al instante qué servicio está caído, si el circuito está abierto o si la cola está saturada.
