======================================================================
1. RESUMEN EJECUTIVO
======================================================================

La arquitectura real del sistema MILPA esta centrada en dos componentes principales:

1. milpa_presenter
   Capa de presentacion y proxy HTTP, escrita en TypeScript con Fastify.

2. milpa_ai_backend
   Capa de negocio e IA, escrita en Python con FastAPI, con pipeline RAG hibrido, OCR, extraccion, indexacion, embeddings y acceso a persistencia.

Alrededor de ese nucleo existe una periferia grande de scripts de diagnostico, verificacion, indexacion, evaluacion y documentacion historica. Esa periferia no invalida la arquitectura central, pero si aumenta de forma importante la complejidad percibida del repositorio.

La arquitectura observable es coherente a nivel logico:
- presenter como punto de entrada web y proxy
- backend como API y motor RAG
- SQLite para metadatos y fragmentos
- ChromaDB para almacenamiento vectorial
- BM25 para recuperacion lexica
- capas auxiliares de OCR, tablas, chunking, NER, feature flags, observabilidad y seguridad

Los principales hallazgos de auditoria son:
- Existe una arquitectura principal clara, pero el repositorio esta disperso.
- El frontend realmente activo es milpa_presenter; la carpeta frontend es legado.
- Hay inconsistencias de persistencia: coexisten referencias a main.db, milpa.db y milpa_knowledge.db.
- El presenter concentra demasiada logica en un unico archivo grande: src/server.ts.
- La documentacion no describe siempre el mismo estado del sistema.
- Existen multiples rutas de acceso y scripts auxiliares que bypassan el camino principal presenter -> backend.

- milpa_presenter/: frontend/proxy principal TypeScript/Fastify.

- Capa API e IA activa: milpa_ai_backend.

- Capa legado: frontend Node/Express/Mongo.

======================================================================
3. FLUJO DE DATOS Y OPERACION REAL
======================================================================

FLUJO PRINCIPAL ESPERADO
1. El usuario entra por el presenter en puerto 8080.
2. El presenter sirve la UI y reenvia llamadas a /ai/* hacia el backend.
3. El backend recibe consultas, uploads, extraccion, reconstruccion de indices y consultas de biblioteca.
4. El backend consulta SQLite para metadatos/fragmentos, BM25 para ranking lexico y ChromaDB para similitud vectorial.
5. El motor RAG fusiona resultados, aplica reranking/sintesis y devuelve respuesta estructurada.
6. El presenter renderiza HTML o pasa JSON hacia la interfaz.

PERSISTENCIA OBSERVABLE
- milpa_ai_backend/data/documents/: documentos y copias persistidas.
- milpa_ai_backend/data/milpa_knowledge.db: base SQLite referida por la configuracion principal.
- milpa_ai_backend/data/main.db: base SQLite referida por migraciones y varios scripts.
- milpa_ai_backend/data/milpa.db: otra base SQLite presente en el repo.
- milpa_ai_backend/data/vector_db/: almacenamiento vectorial.
- milpa_ai_backend/data/bm25_idx/ y milpa_ai_backend/data/bm25_index/: indices lexicos.
- milpa_ai_backend/data/tmp/: temporales.
- data/ en la raiz: tambien contiene db/indices, lo que agrega ambiguedad operacional.

ORQUESTACION
- docker-compose.yml define los servicios clamav, ai, presenter, prometheus y grafana.
- La narrativa arquitectonica indica un punto de entrada principal en 8080, pero Compose tambien publica el backend en 8000.

======================================================================
5. HALLAZGOS DE AUDITORIA
======================================================================

5.1. Hallazgo principal: el nucleo esta claro, el repositorio no.
La arquitectura central presenter + backend + persistencia hibrida es reconocible y defendible. La complejidad mayor viene de la periferia: scripts, rutas de datos alternativas, documentacion historica y componentes heredados.

5.2. Hallazgo estructural: frontend activo vs frontend legado.
- Activo: milpa_presenter.
- Legado: frontend.
La carpeta frontend contiene un stack distinto (Express, MongoDB, Socket.IO, JWT, Mongoose) que ya no coincide con la arquitectura principal actual. Su presencia aumenta el ruido tecnico.

5.3. Hallazgo de persistencia: multiples bases SQLite coexistiendo.
Se detectaron referencias a:
- data/milpa_knowledge.db
- data/main.db
- data/milpa.db

Evidencia observada:
- core/config.py usa por defecto data/milpa_knowledge.db.
- yoyo.ini apunta a data/main.db.
- core/config_flags/feature_flags.py inicializa por defecto contra data/main.db.
- varios scripts de raiz consultan milpa_knowledge.db.
- otros scripts/documentos refieren milpa.db.
- dentro de milpa_ai_backend/data existen fisicamente main.db, milpa.db y milpa_knowledge.db.

Impacto:
- posibilidad de leer/escribir estados distintos segun el script.
- migraciones y feature flags pueden no residir en la misma base que usa el runtime principal.
- debugging mas confuso.

5.4. Hallazgo de modularidad: presenter monolitico.
milpa_presenter/src/server.ts concentra muchas responsabilidades:
- arranque del servidor
- proxy al backend
- paginas HTML
- JavaScript embebido
- colas/concurrencia
- chequeos de runtime
- sanitizacion y utilidades de interfaz

Impacto:
- alto acoplamiento
- mantenimiento mas costoso
- baja separacion de concerns en la capa web

5.5. Hallazgo operativo: la narrativa de punto de entrada unico no esta completamente cerrada.
Aunque la documentacion presenta al presenter como punto de entrada unico, Compose publica tambien el backend en 8000 y muchos scripts trabajan directamente contra el backend. Esto genera dos rutas de uso reales: via presenter o via backend directo.

5.6. Hallazgo documental: deriva de documentacion.
Hay abundante documentacion, pero no toda describe el mismo estado del sistema. Esto obliga a leer varias piezas y reconciliarlas manualmente.

5.7. Hallazgo de datos: persistencia duplicada en la raiz y dentro del backend.
Se observan artefactos de datos tanto en data/ (raiz) como en milpa_ai_backend/data/. Esto agrega ambiguedad sobre que snapshot o ubicacion es la operativa/canonica en cada contexto.

5.8. Hallazgo de deuda tecnica: blue-green parcialmente integrado.
Existe un modulo blue_green.ts con logica de canary/blue-green, pero no emerge como eje dominante del flujo principal observado. Eso sugiere funcionalidad planificada o parcialmente incorporada.

5.9. Hallazgo de empaquetado: backend con senales de fragilidad operativa.
La coexistencia de varios scripts que operan por ruta directa, mas configuraciones de DB no unificadas, sugiere que parte del funcionamiento depende de convenciones del entorno y no de una unica capa de composicion rigurosa.

======================================================================
6. CONCLUSION TECNICA
======================================================================

Conclusion principal
El sistema MILPA si tiene una arquitectura principal identificable y relativamente bien separada en lo conceptual:
- presenter para web/proxy
- backend para IA y negocio
- SQLite + BM25 + ChromaDB + filesystem para persistencia/retrieval

Sin embargo, la experiencia de mantenimiento del repositorio queda deteriorada por:
- coexistencia de codigo legado
- proliferacion de scripts auxiliares
- varias rutas de base de datos
- documentacion de etapas diferentes
- concentracion de UI y proxy en un archivo grande

Si se toma solo el nucleo operativo, la arquitectura es de dos capas con retrieval hibrido. Si se toma el repositorio completo como producto de ingenieria, el principal problema no es la ausencia de arquitectura, sino la falta de consolidacion.

======================================================================
7. SINTESIS CORTA PARA DECISION
======================================================================

- Arquitectura activa real: milpa_presenter + milpa_ai_backend.
- Motor central: RAG hibrido con BM25 + embeddings + enriquecimiento por entidades.
- Persistencia real observada: varias DB SQLite coexistentes, vector store y filesystem documental.
- Mayor riesgo tecnico: inconsistencia de rutas/DB y dispersion del repositorio.
- Mayor riesgo de mantenimiento: presenter monolitico y legado frontend no removido.
- Estado general: nucleo funcional coherente, repositorio operativamente heterogeneo.

======================================================================
9. FLUJO COMPLETO DE INGESTA DE DATOS (upload -> indice)
======================================================================

Este apartado traza la cadena de llamadas real desde que un usuario sube un archivo hasta que queda indexado y buscable.

9.1. Paso 1: Upload del documento
  Punto de entrada: POST /api/documents/upload
  Archivo: milpa_ai_backend/api/endpoints.py -> upload_document()
  Validaciones antes de persistir:
    - Tipo MIME permitido: application/pdf, .docx, text/plain.
    - Clasificacion obligatoria: Publico, Interno o Restringido.
    - Licencia obligatoria: institutional, public_domain, permitted, normative.
    - Tamano maximo: MAX_UPLOAD_MB (por defecto 25 MB).

9.2. Paso 2: Persistencia del original
  Llamada: persist_original(file, filename) en core/logic/ingestion.py
  Proceso:
    1. Resuelve la ruta absoluta de data/documents/.
    2. Sanitiza el nombre del archivo (_sanitize_filename).
    3. Antepone un timestamp para unicidad.
    4. Escribe el archivo a disco en streaming (no carga todo en memoria).
    5. Calcula el SHA-256 del archivo guardado -> este hash se usa como doc_id.
  Salida: (doc_id, stored_path)

9.3. Paso 3: Escaneo antivirus
  Llamada: scan_file_strict(stored_path) en core/security/av.py
  Proceso:
    1. Conecta a ClamAV por TCP (clamav:3310) via clamd.ClamdNetworkSocket.
    2. Ejecuta SCAN por ruta del filesystem (clamd accede al archivo via volumen compartido).
    3. Ejecuta INSTREAM enviando el contenido del archivo por stream TCP.
    4. Si cualquiera de los dos metodos detecta malware -> raise AntivirusError.
    5. Si ClamAV no esta disponible y AV_OPTIONAL=true -> continua con warning.
  Razon de la doble verificacion: SCAN puede fallar si /tmp es read-only; INSTREAM es independiente del filesystem. Juntos cubren ambos vectores.

9.4. Paso 4: Registro en base de datos
  Ubicacion: endpoints.py, despues del escaneo exitoso.
  Operaciones SQL:
    INSERT INTO docs (doc_id, title, author, year, source, hash, license, lang_original, classification, created_at, stored_path)
    INSERT INTO licenses (si aplica, segun el flujo de metadatos)
  Retorna al cliente: {doc_id, stored_path, license, classification, title, author, year}

9.5. Paso 5: Extraccion de contenido
  Punto de entrada: POST /api/documents/{doc_id}/extract
  Archivo: milpa_ai_backend/api/endpoints.py -> extract_endpoint()
  Llamada interna: extract_document(conn, doc_id, pdf_path, ...) en core/logic/extract.py
  Proceso por pagina:
    1. Abre el PDF con PyMuPDF (fitz.open).
    2. Extrae texto nativo: page.get_text("text").
    3. Evalua si necesita OCR: _page_needs_ocr(text, threshold=40).
       Criterio: si el texto nativo tiene menos de 40 caracteres utiles, se considera escaneado.
    4. Si necesita OCR: renderiza la pagina como imagen y ejecuta pytesseract.image_to_string(img, lang="spa+eng").
    5. Resultado: texto por pagina con fuente marcada ("native" u "ocr").

9.6. Paso 6: Fragmentacion (chunking)
  Llamada: _chunk_text(text, chunk_size=1200) en core/logic/extract.py
  Proceso:
    1. Divide el texto por doble salto de linea (parrafos).
    2. Si un parrafo excede chunk_size, lo subdivide por palabras.
    3. Genera fragmentos de ~1200 caracteres con limites semanticos.
  Resultado: lista de fragmentos (strings).

9.7. Paso 7: Enriquecimiento de entidades (NER)
  Llamada: extract_entities(text) en core/logic/enrichment.py
  Proceso:
    1. Normaliza el texto (acentos, minusculas).
    2. Carga la taxonomia agricola (CSV de cultivos, plagas, nutrientes + JSON de sinonimos).
    3. Ejecuta NER por diccionario (_dict_ner): busca coincidencias exactas de terminos del dominio.
    4. Aumenta con spaCy (_augment_with_spacy): extrae entidades adicionales con modelo NLP.
    5. Normaliza sinonimos (e.g., "elote" -> "maiz").
    6. Clasifica el fragmento por tipo: RECOMENDACION, ADVERTENCIA, DATO, etc.
  Resultado: (lista de Entity, dict de sinonimos aplicados, texto normalizado)

9.8. Paso 8: Insercion de fragmentos en SQLite
  Llamada: _insert_fragment(cur, doc_id, page_no, chunk, source) en core/logic/extract.py
  Proceso:
    1. Genera un fragment_id con uuid().
    2. Ejecuta extract_entities(text) para obtener entidades JSON.
    3. INSERT INTO fragments (fragment_id, doc_id, fragment_uid, page_start, page_end, text, source, entities, created_at).

9.9. Paso 9: Extraccion de tablas
  Llamada: _try_camelot(pdf_path) en core/logic/tables.py
  Proceso:
    1. Intenta Camelot con flavor="lattice" (tablas con bordes visibles).
    2. Si falla o no encuentra tablas, intenta flavor="stream" (tablas sin bordes).
    3. Si Camelot no esta disponible, usa _fallback_detect_tables_pymupdf() (heuristica por lineas/formas).
    4. Para cada tabla detectada:
       - _insert_table(cur, doc_id, page, csv_text, bbox, schema_obj)
       - _insert_table_cells(cur, table_id, dataframe)

9.10. Paso 10: Indexacion en BM25 y vector store
  Punto de entrada: POST /api/index/rebuild  o  rebuild_indexes() en api/rag.py
  Proceso:
    1. Lee todos los fragmentos de SQLite.
    2. Para BM25 (core/logic/bm25.py):
       - BM25Index.index_many(records): indexa texto + labels + entidades de cada fragmento.
       - Backends en cascada: Tantivy (preferido) -> Whoosh -> memoria.
    3. Para vector store (core/logic/vectordb.py):
       - EmbeddingModel.encode(textos) -> vectores de 384 dims (MiniLM) o 1024 dims (BGE-m3 segun flag).
       - VectorStore.upsert(ids, embeddings, metadatas): persiste en ChromaDB.
    4. Al finalizar, el sistema puede responder consultas hibridas.

======================================================================
10. FLUJO COMPLETO DE CONSULTA RAG (query -> respuesta)
======================================================================

10.1. Punto de entrada
  POST /api/query
  Archivo: milpa_ai_backend/api/rag.py -> query_rag()
  Parametros de entrada (QueryRequest):
    query:          texto de la consulta del usuario (obligatorio)
    k:              numero de fragmentos a devolver (defecto 8)
    mode:           "hybrid" | "dense" | "lex" (defecto "hybrid")
    labels_filter:  lista opcional de etiquetas para filtrar

10.2. Inicializacion lazy de componentes
  Llamada: get_retriever() en rag.py
  Proceso:
    1. Si no existen, instancia los singletons globales:
       _embedder    = EmbeddingModel()     -> carga paraphrase-multilingual-MiniLM-L12-v2
       _vector_store = VectorStore()       -> conecta a ChromaDB persistente
       _bm25_index  = BM25Index()          -> carga indice Tantivy/Whoosh
       _retriever   = HybridRetriever(vector_store, bm25_index, embedder)
    2. Los singletons se reusan en consultas posteriores.

10.3. Busqueda segun modo

  MODO "dense" (solo vectorial):
    retriever.dense_search(query, k)
      -> embedder.encode(query) -> vector de consulta
      -> vector_store.query(query_emb=vector, k=k)
      -> ChromaDB cosine similarity
      -> retorna [{fragment_id, score, metadata}]

  MODO "lex" (solo lexico):
    retriever.lex_search(query, topk=k*10, labels_filter)
      -> bm25_index.search(query, topk, labels_filter)
      -> Tantivy/Whoosh BM25 scoring
      -> retorna top-k fragmentos

  MODO "hybrid" (defecto, combinado):
    1. vec_hits = dense_search(query, k=K_RETRIEVE)
    2. bm25_hits = lex_search(query, topk=BM25_TOPK)
       Donde BM25_TOPK = 100 por defecto
    3. fused = reciprocal_rank_fusion(vec_hits, bm25_hits, K=RRF_K)
       Formula: RRF_score(d) = sum( 1 / (K + rank_i) ) para cada ranking donde aparece d
       K = 60 (parametro de suavizado)
    4. Resultado: lista fusionada ordenada por score RRF

10.4. Reranking (solo en modo hybrid)
  Llamada: rerank_by_term_coverage(query, hits, fragment_texts)
  Proceso:
    1. Extrae terminos del query (longitud > 2, excluye stopwords).
    2. Para cada hit: coverage_ratio = terminos_matcheados / total_terminos_query.
    3. boosted_score = score_original * (1 + coverage_ratio * 0.5).
    4. Reordena por boosted_score.
    5. Limita a k resultados finales.

10.5. Evaluacion de evidencia insuficiente
  Llamada: insufficient_evidence(query, hits) en rag_engine.py
  Umbrales aplicados (clase Thresholds):
    k_min                = 3      (minimo de fragmentos tras filtrado)
    min_sources          = 1      (minimo de documentos distintos)
    entity_coverage_min  = 0.10   (cobertura minima de entidades query vs fragmentos)
    mean_score_min       = 0.010  (puntaje promedio minimo de fragmentos)
    min_fragment_score   = 0.003  (puntaje minimo individual por fragmento)
  Proceso:
    1. Filtra fragmentos con score < min_fragment_score.
    2. Si quedan < k_min fragmentos -> evidencia insuficiente.
    3. Extrae entidades del query y calcula entity_coverage vs fragmentos recuperados.
    4. Si entity_coverage < entity_coverage_min -> evidencia insuficiente.
    5. Si evidencia insuficiente:
       - build_insufficient_response(query, diagnostics) genera respuesta de fallback.
       - Retorna QueryResponse con insufficient_evidence=True.

10.6. Carga de metadatos de fragmentos
  Consulta SQL:
    SELECT f.text, f.doc_id, f.page_start, f.page_end, f.entities, d.title
    FROM fragments f LEFT JOIN docs d ON f.doc_id = d.doc_id
    WHERE f.fragment_id IN (...)
  Construye objetos FragmentResponse con texto, metadata y ubicacion.

10.7. Generacion de respuesta y citaciones
  Llamada: compose_answer(query, fragments, max_length=500) en synthesis.py
  Proceso:
    1. extract_sentences(texto): tokeniza el texto de cada fragmento en oraciones.
    2. Concatena los fragmentos relevantes como contexto.
    3. compute_faithfulness(respuesta, fragmentos): calcula que proporcion de la respuesta esta soportada por los fragmentos.
    4. build_citation(fragment, idx): para cada fragmento, construye un objeto citacion con:
       - doc_id, page, bbox (coordenadas PDF), table_id (si aplica)
       - link href para click-through en el viewer (/doc/:id?page=N)
    5. sanitize_html(respuesta_html): limpia HTML de la respuesta.
  Resultado: {respuesta_html, citas[], faithfulness}

10.8. Respuesta final al cliente
  Estructura QueryResponse:
    query:                texto original de la consulta
    fragments:            lista de FragmentResponse (texto, doc_id, page, score)
    total_retrieved:      total de fragmentos considerados
    mode:                 modo usado (hybrid/dense/lex)
    answer:               texto/HTML de la respuesta generada
    answer_mode:          modo de generacion (local/concat)
    citations:            lista de citaciones con metadata
    insufficient_evidence: bool indicando si la evidencia fue insuficiente

10.9. Diagrama de conexiones de consulta

  Usuario (navegador)
      |
      | POST /ai/api/query  {query: "fertilizacion de maiz", k: 5, mode: "hybrid"}

  milpa_presenter (Fastify :8080)

12. MECANICA DEL PROXY PRESENTER -> BACKEND
======================================================================

El presenter actua como proxy inverso para todas las rutas /ai/*. La implementacion esta en milpa_presenter/src/server.ts.

12.1. Ruta: ALL /ai/*
  Captura cualquier metodo HTTP (GET, POST, PUT, DELETE, PATCH) hacia cualquier subruta de /ai/.

12.2. Proceso paso a paso
  1. Extrae el sufijo: quita el prefijo "/ai/" de la URL y conserva path + query string.
     Ejemplo: /ai/library?page=2 -> sufijo = "library?page=2"

  2. Construye la URL destino: new URL(sufijo, config.IA_URL)
     Donde config.IA_URL = "http://ai:8000" en Docker (nombre DNS del servicio).
     Resultado: "http://ai:8000/library?page=2"

  3. Clona headers: copia headers del request original, elimina hop-by-hop (host, content-length).

  4. Prepara el cuerpo:
     - GET/HEAD: sin cuerpo.
     - POST/PUT/PATCH: si es string/Buffer, lo usa directo. Si es objeto, lo serializa con JSON.stringify y agrega Content-Type: application/json.

  5. Control de concurrencia (Scheduler):
     - Verifica que la cola no este llena (QUEUE_CAPACITY = 64).
     - Si esta llena -> retorna HTTP 429 con header Retry-After: 2.
     - Si hay espacio, encola la peticion.
     - Maximo de peticiones simultaneas: MAX_CONCURRENCY = 8.

  6. Circuit breaker (CircuitBreaker):
     - Si el circuito esta OPEN -> retorna HTTP 503.
     - Si esta CLOSED o HALF_OPEN -> permite la peticion.
     - Tras exito: recordSuccess().
     - Tras fallo: recordFailure(). Si acumula suficientes fallos -> abre el circuito por CIRCUIT_OPEN_SECS = 30s.

  7. Ejecuta fetch() hacia el backend con timeout (AbortController + signal).

  8. Copia la respuesta:
     - Codigo HTTP del backend.
     - Headers del backend (excepto content-length y content-encoding).
     - Cuerpo como stream.

  9. Manejo de errores:
     - Timeout del fetch -> HTTP 504 Gateway Timeout.
     - Error de red/conexion -> HTTP 502 Bad Gateway.

12.3. Metricas registradas en cada request de proxy
  - milpa_proxy_total: incrementa en cada request.
  - milpa_proxy_429_total: incrementa si la cola esta llena.
  - milpa_proxy_timeout_total: incrementa en timeout.
  - milpa_circuit_open_total: incrementa cuando el circuito se abre.
  - milpa_proxy_latency_ms: registra latencia en histograma.
  - milpa_queue_in_flight: gauge de peticiones en vuelo.
  - milpa_queue_depth: gauge de peticiones esperando.

14. CAPA DE SEGURIDAD DETALLADA
======================================================================

14.1. Backend: antivirus ClamAV
  Archivo: milpa_ai_backend/core/security/av.py
  Protocolo: doble verificacion SCAN + INSTREAM.
    SCAN: ClamAV accede al archivo directamente por ruta del filesystem (volumen compartido).
    INSTREAM: el backend envia el contenido del archivo por TCP stream a ClamAV.
    Si cualquiera detecta malware -> AntivirusError -> HTTP 400 al cliente.
    Si ClamAV no esta disponible y AV_OPTIONAL=true -> el upload continua con warning en log.
  Configuracion de ClamAV:
    - clamd.conf montado desde ./clamav/clamd.conf
    - TCPSocket en 3310, sin LocalSocket
    - /tmp montado como tmpfs RW (necesario para working dir de clamd)

14.2. Presenter: headers HTTP de seguridad
  Archivo: milpa_presenter/src/security/headers.ts
  Headers inyectados en cada respuesta:
    X-Content-Type-Options:    nosniff
    X-Frame-Options:           DENY
    Referrer-Policy:           no-referrer
    Strict-Transport-Security: max-age=31536000; includeSubDomains
    Content-Security-Policy:
      default-src 'none';
      script-src 'unsafe-inline';
      img-src 'self' data:;
      style-src 'self' 'unsafe-inline';
      font-src 'self';
      connect-src 'self';
      frame-ancestors 'none';
      base-uri 'none';
      form-action 'self'

14.3. Presenter: sanitizacion de HTML
  Archivo: milpa_presenter/src/security/sanitize.ts
  Tags permitidos: a, em, strong, ul, ol, li, p, br, code, pre.
  Atributos permitidos en <a>: href, data-cite, data-page, data-bbox, data-table, data-row, data-col.
  Esquemas bloqueados: todos (http, https, mailto, etc.).
  Solo se preservan URLs internas (prefijos: /, #, ?, ./, ../).
  URLs externas se reemplazan por "#".
  Tags no permitidos: se descartan (no se escapan).

14.4. Presenter: rate limiting
  Configurado en server.ts al registrar fastify-rate-limit.
  Limite: 60 requests por minuto por IP.
  Respuesta al exceder: HTTP 429 Too Many Requests.

14.5. Presenter: circuit breaker
  Archivo: milpa_presenter/src/runtime/circuit.ts
  Estados: CLOSED (normal) -> OPEN (fallo) -> HALF_OPEN (prueba).
  Abre despues de N fallos consecutivos o latencia excesiva.
  Periodo abierto: CIRCUIT_OPEN_SECS = 30 segundos.
  Permite una request de prueba en HALF_OPEN; si tiene exito, vuelve a CLOSED.

14.6. Frontend legado: seguridad (observacion)
  Archivo: frontend/routes/api.js
  JWT_SECRET = '12345' (hardcodeado en el codigo fuente).
  Hallazgo: secreto JWT trivial. Si el frontend legado llegara a reactivarse, seria una vulnerabilidad critica.

16. FEATURE FLAGS: DETALLE Y CONSUMO
======================================================================

16.1. Almacenamiento
  Tabla: feature_flags en SQLite.
  Cargados en memoria al iniciar (cache en FeatureFlags._cache).
  Recargables en caliente con reload() sin reiniciar el servicio.
  Modificables por API: PUT /admin/feature-flags/{name}

16.2. Clase gestora
  Archivo: milpa_ai_backend/core/config_flags/feature_flags.py
  Clase: FeatureFlags
  Singleton: feature_flags = FeatureFlags()
  Metodos:
    is_enabled(flag_name, default) -> bool
    get_config(flag_name, key, default) -> Any
    reload() -> None
    set_flag(flag_name, enabled, config) -> None

16.3. Helpers de acceso rapido
    is_reranker_enabled()           -> True si RERANKER_ENABLED esta habilitado
    get_embeddings_model()          -> nombre del modelo (defecto "BAAI/bge-m3")
    get_rag_mode()                  -> dict con mode, bm25_weight, vector_weight
    is_blue_green_v2_enabled()      -> (enabled: bool, rollout_percent: int)

16.4. Flags definidos y su efecto
    RERANKER_ENABLED:
      Controla si se aplica reranking con cross-encoder despues de la fusion RRF.
      Si esta deshabilitado, la respuesta se da con el ranking directo de RRF.

    EMBEDDINGS_MODEL:
      Define el modelo de embeddings a usar. Puede cambiar entre MiniLM y BGE-m3 sin redeploy.
      Afecta la dimensionalidad del vector store.

    RAG_MODE:
      Define pesos relativos de BM25 vs vectorial en la fusion.
      Permite ajustar la estrategia de retrieval sin cambiar codigo.

    TAXONOMY_VERSION:
      Version de la taxonomia agricola usada para NER.
      Permite cargar nuevas versiones de listas de cultivos/plagas/nutrientes.

    BLUE_GREEN_V2_ENABLED:
      Controla el rollout gradual de una version v2 de la UI.
      rollout_percent define que porcentaje de sesiones se redirige a /ui/v2.
      Por defecto deshabilitado (0%).

    TABLE_EXTRACTION_MODE:
      Controla las estrategias de Camelot: auto, lattice, stream o ambas.

    ENRICHMENT_ENABLED:
      Controla si se ejecuta NER y clasificacion sobre los fragmentos durante extraccion.

    AV_STRICT_MODE:
      Controla si se ejecutan ambos modos de escaneo (SCAN + INSTREAM) o solo uno.

    OCR_ENABLED:
      Controla si se aplica OCR a paginas con poco texto nativo.
      Configura idioma de Tesseract y umbral minimo de texto.

  Funcion: iniciarServidor()
  Proceso:
    1. Conecta a MongoDB (process.env.MONGODB_URI).
    2. Levanta Express en el puerto configurado.
    3. Registra rutas estaticas desde frontend/MILPA/.
    4. Configura Socket.IO con autenticacion JWT en el handshake.

17.3. Colecciones MongoDB
  users:         username, password (hash bcrypt), createdAt
  messages:      user (ref a users), texto, type, createdAt
  notifications: type, title, message, createdAt

17.4. Endpoints REST
  POST /api/auth/register  -> crea usuario (valida unicidad, hashea password con bcrypt)
  POST /api/auth/login     -> autentica, retorna JWT (expira en 24h)
  Middleware verificarToken -> extrae "Bearer <token>" de Authorization, verifica con jwt.verify

17.5. Socket.IO: eventos
  connection:            se autentica via socket.handshake.auth.token
  request_initial_data:  cliente pide historial -> servidor emite data_update con todos los Message
  mensaje_chat:          cliente envia {texto} -> guarda en DB -> broadcast nuevo_mensaje a todos
  push_notification:     al conectar, servidor emite las ultimas 3 Notification

17.6. Funcion responderComoBot(mensaje, usuarioId, usuarioNombre)
  Genera una respuesta automatica del sistema al mensaje del usuario.
  Guarda el mensaje del bot en la coleccion messages.
  Emite nuevo_mensaje por Socket.IO.

17.7. Paginas web del frontend legado

17.10. Hallazgo de seguridad: JWT_SECRET hardcodeado
  En frontend/routes/api.js linea 7:

  Este secreto es trivial y esta directamente en el codigo fuente.
  Si el frontend legado se reactivara, cualquier persona podria forjar tokens JWT validos.

20. CONCLUSION AMPLIADA
======================================================================

20.1. Sobre completitud del sistema
El sistema MILPA implementa un pipeline RAG completo de punta a punta:
- Ingesta: upload -> antivirus -> persistencia -> extraccion (PDF nativo + OCR + tablas) -> chunking -> NER -> indexacion BM25 + vectorial.
- Consulta: query -> busqueda hibrida (BM25 + embeddings) -> fusion RRF -> reranking -> evaluacion de evidencia -> generacion de respuesta -> citaciones -> respuesta al usuario.
- Presentacion: paginas web SSR con biblioteca, detalle documental, interfaz de consulta y verificacion de sistema.
- Operacion: Docker Compose con hardening, antivirus, observabilidad (Prometheus + Grafana + OpenTelemetry) y feature flags dinamicos.

20.2. Sobre los vacios verificados
Ninguno de los componentes principales carece de implementacion funcional detectable por lectura estatica. Los vacios son operativos, no arquitectonicos:
- La inconsistencia de bases de datos (milpa_knowledge.db vs main.db) es el problema mas concreto.
- El presenter monolitico dificulta mantenimiento pero funciona.
- El frontend legado es ruido, no un vacio.
- Blue-green esta parcialmente integrado pero los componentes existen.

20.3. Sobre descriptividad de procesos
Esta auditoria ampliada incluye:
- Esquema de base de datos con todas las tablas, columnas, tipos e indices.
- Cadena completa de ingesta paso a paso con funciones y archivos.
- Cadena completa de consulta RAG paso a paso con parametros y umbrales.
- Descripcion de cada pagina web servida y las llamadas fetch() que realiza.
- Mecanica exacta del proxy con control de concurrencia y circuit breaker.
- Red Docker con volumenes, variables de entorno y hardening.
- Detalle de seguridad: ClamAV, headers, sanitizacion, rate limiting.
- Observabilidad: OpenTelemetry, Prometheus, Grafana con metricas nombradas.
- Feature flags: listado completo con efecto de cada uno.
- Frontend legado: MongoDB, Socket.IO, JWT, paginas y hallazgos de seguridad.
- Mapa global de conexiones entre todos los componentes.
